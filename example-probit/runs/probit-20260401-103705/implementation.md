## Key Implementation Decisions

1. **C++ function naming**: C++ exports named `probit_mle_cpp`, `probit_gibbs_cpp`, `probit_mh_cpp` to avoid collision with R wrapper functions `probit_mle`, `probit_gibbs`, `probit_mh`.

2. **Numerical stability**: All log-CDF computations use `safe_log_Phi` (R::pnorm with log.p=TRUE). Newton step solved via `arma::solve` (not matrix inversion). Lambda guard at Phi < 1e-300 uses asymptotic |q_i * eta_i| approximation. Hessian diagonal d_i clamped to 1e-10 minimum. MH acceptance computed entirely on log scale.

3. **Truncated normal sampling**: Inverse CDF method with R::pnorm/R::qnorm. Probability clamped to [1e-15, 1-1e-15] to prevent qnorm returning +/-Inf. Handles infinite bounds via R_PosInf/R_NegInf.

4. **MH initialization**: Defaults to MLE estimate (calls `probit_mle_cpp` internally) when no init provided, as specified.

5. **Hessian computation**: Uses vectorized `X.each_col() % d` for efficient formation of X'DX without constructing the full diagonal matrix.

6. **CRAN compliance**: TRUE/FALSE throughout, no hardcoded set.seed in functions, message() not print(), MCMC examples wrapped in `\donttest{}`, Authors@R format, 2+ sentence Description.

7. **Pre-existing file**: `R/simulate_probit.R` and its export `run_probit_simulation` were already present in the repo (from a prior workflow step). These were left untouched.

## BLOCK Fix: MH Acceptance Rate Tuning (Respawn)

**Problem**: Tester BLOCK on criterion C3 -- MH acceptance rate was ~0.70 across all sample sizes. The required range is [0.20, 0.50]. The default `scale = 1.0` produced proposals too conservative (too close to current value), leading to excessive acceptance.

**Root cause**: The proposal covariance `scale^2 * (X'X)^{-1}` with `scale = 1.0` generates proposals with small variance relative to the posterior, so nearly all proposals are accepted. The standard Roberts-Rosenthal optimal scaling for random-walk MH suggests a larger scale factor.

**Fix applied**: Changed default `scale` from `1.0` to `2.4` in three locations:
1. `src/probit_mh.cpp` line 35: C++ default parameter `double scale = 2.4`
2. `R/probit_mh.R` line 45: R wrapper default `scale = 2.4`
3. `R/probit_mh.R` roxygen2 `@param scale` documentation updated to reflect new default

**Files modified**:
- `src/probit_mh.cpp` (default parameter value)
- `R/probit_mh.R` (default parameter value + roxygen2 docs)
- `man/probit_mh.Rd` (regenerated by roxygen2)
- `src/RcppExports.cpp` (regenerated by Rcpp::compileAttributes)
- `R/RcppExports.R` (regenerated by Rcpp::compileAttributes)

**Verification results** (acceptance rates with `scale = 2.4`):
| N | Acceptance Rate |
| --- | --- |
| 200 | 0.3929 |
| 500 | 0.4059 |
| 1000 | 0.4007 |
| 2000 | 0.3908 |

All sample sizes produce acceptance rates in the 0.39-0.41 range, well within the C3 target [0.20, 0.50]. The rates are stable across sample sizes because `(X'X)^{-1}` scales naturally with N.

**Algorithm unchanged**: Only the default parameter value was modified. The MH algorithm, log-posterior computation, and proposal mechanism are identical.

## BLOCK Fix 2: Roxygen2 Percent Escape + Simulation Scale Mismatch (Respawn 2)

### Problem A: roxygen2 `\%` escape in `R/probit_mh.R`

Line 16 of `R/probit_mh.R` contained `20-50\%.` in the `@param scale` roxygen2 description. The `\%` escape in roxygen2 produces `\\%` in the generated `.Rd` file, which corrupts the `\arguments` section and causes `R CMD check` to emit a warning or error about malformed Rd content.

**Fix**: Changed `20-50\%` to `20-50 percent` (plain text, avoids all escaping issues).

**File modified**: `R/probit_mh.R` (line 16)

### Problem B: Simulation scripts hardcode `scale = 1.0`

Both `inst/simulation/run_simulation.R` (line 31) and `R/simulate_probit.R` (line 59) had `scale = 1.0` hardcoded in the MH estimator configuration. 1. `inst/simulation/run_simulation.R` line 31: `mh_args` list
2. `R/simulate_probit.R` line 59: `mh_args` list inside `run_probit_simulation()`
