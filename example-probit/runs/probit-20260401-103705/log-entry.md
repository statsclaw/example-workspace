# 2026-04-01 -- exampleProbit Initial Build with Monte Carlo Simulation

> Run: `probit-20260401-103705` | Profile: r-package | Verdict: PASS

## What Changed

Built the `exampleProbit` R package from scratch, implementing three probit model estimation methods (MLE via Newton-Raphson, Bayesian Gibbs sampler via Albert-Chib data augmentation, and random-walk Metropolis-Hastings) in C++ via Rcpp/RcppArmadillo. Created R wrapper functions with roxygen2 documentation, 28 unit tests, and a full Monte Carlo simulation study comparing all three methods across four sample sizes (N = 200, 500, 1000, 5000) with R = 500 replications. All seven acceptance criteria (C1-C7) pass.

## Design Decisions

1. **Default MH scale = 2.4**: The original spec used scale = 1.0, which produced acceptance rates around 70 percent (proposals too close to current state). The Roberts-Rosenthal optimal scaling theory suggests larger values. Scale 2.4 yields approximately 40 percent acceptance across all sample sizes, well within the 20-50 percent target. The acceptance rate is stable across N because (X'X)^{-1} scales naturally.

2. **Cholesky precomputation in Gibbs**: Sigma_tilde = (Sigma_0^{-1} + X'X)^{-1} and its Cholesky factor L depend only on X and the prior, not on the MCMC state. Computing them once before the loop avoids redundant O(k^3) operations per iteration.

3. **MLE warm-start for MH**: probit_mh_cpp calls probit_mle_cpp internally when no init vector is provided. This places the chain near the posterior mode from the first iteration, reducing the burn-in needed for convergence.

4. **Log-space CDF computation**: safe_log_Phi uses R::pnorm with log.p=TRUE rather than computing Phi(x) and taking log. This avoids catastrophic cancellation for extreme x values where Phi(x) is near 0 or 1.

5. **Newton step via solve, not inversion**: The Newton update uses arma::solve(H, g) instead of arma::inv(H) * g. This is numerically more stable and avoids forming the explicit inverse. The inverse is only computed once at the end (via inv_sympd) for the variance-covariance matrix.

6. **C++ _cpp suffix naming**: C++ exports use the _cpp suffix (probit_mle_cpp, etc.) so that R wrapper functions can use the clean names (probit_mle, etc.) without collision, keeping the user-facing API clean.

7. **Sequential simulation**: The Monte Carlo study runs sequentially (no parallelization) to comply with CRAN's 2-core limit for tests and examples.
